# Waste Management Monitoring & Heatmap System — Backend Engineering Specification

## Project Purpose

The backend system powers a civic-tech environmental monitoring platform focused on waste management, drainage blockage detection, environmental hotspot monitoring, and cleanup workflow management.

The system allows:

* citizens to submit waste-related reports
* administrators to monitor and manage incidents
* frontend applications to retrieve environmental heatmaps and alerts
* operational authorities to track cleanup progress

The backend is responsible for:

* API communication
* database management
* report verification
* heatmap data preparation
* severity analysis
* operational state tracking
* environmental analytics

---

# CORE BACKEND OBJECTIVES

The backend must:

1. Receive waste reports from frontend clients
2. Store reports safely in the database
3. Validate and verify incoming data
4. Generate structured data for heatmaps
5. Maintain incident lifecycle states
6. Provide APIs for frontend and admin dashboard
7. Handle analytics and clustering logic
8. Support future AI integration

---

# SYSTEM ARCHITECTURE

```text
Frontend User App
       ↓
HTTP API Requests
       ↓
FastAPI Backend
       ↓
Verification + Processing Logic
       ↓
SQLite Database
       ↓
Heatmap + Analytics Engine
       ↓
Admin Dashboard APIs
```

---

# SECURITY & AUTHENTICATION

*   **Public Access:** `POST /report`, `GET /reports`, `GET /heatmap`, and `GET /alerts` are public.
*   **Admin Access:** `GET /admin/reports`, `GET /admin/analytics`, and `PATCH /report/{id}` require a **Bearer Token** (verified against `ADMIN_API_KEY` env var).
*   **CORS:** Explicitly allowed for Hugging Face Space domains and local development. Wildcards are prohibited due to file upload handling.

# DATABASE DESIGN

## Reports Table

The reports table stores all waste incident information.

| Field            | Type     | Purpose                  |
| ---------------- | -------- | ------------------------ |
| id               | Integer  | Unique report ID         |
| reporter_id      | String   | Anonymous ID / User ID    |
| image_path       | String   | Uploaded image location  |
| image_hash       | String   | MD5/SHA hash for duplicates|
| latitude         | Float    | GPS latitude             |
| longitude        | Float    | GPS longitude            |
| location_name    | String   | Human-readable location  |
| description      | Text     | User explanation         |
| waste_type       | String   | Waste classification     |
| severity         | String   | Low/Medium/High/Critical |
| severity_score   | Integer  | Numeric intensity        |
| state            | String   | Incident lifecycle state |
| confidence_score | Integer  | Verification confidence  |
| created_at       | DateTime | Report timestamp         |
| updated_at       | DateTime | Last modification        |
| is_duplicate     | Boolean  | Duplicate detection      |

### Severity to Score Mapping

| Severity | Score |
| -------- | ----- |
| Low      | 20    |
| Medium   | 50    |
| High     | 80    |
| Critical | 100   |

*Note: `severity_score` is automatically updated via `SEVERITY_SCORES` constant on write.*

# INCIDENT STATE SYSTEM

The backend must implement a report lifecycle.

## Allowed States

| State         | Meaning                  |
| ------------- | ------------------------ |
| reported      | Newly submitted report   |
| verified      | Report confirmed valid   |
| investigating | Admin reviewing incident |
| assigned      | Cleanup team assigned    |
| resolved      | Waste issue cleaned      |
| rejected      | Invalid/fake report      |
| expired       | Report too old           |

### Legal State Transitions

| From State    | Allowed Next States             |
| ------------- | ------------------------------- |
| reported      | verified, rejected, expired      |
| verified      | investigating, rejected         |
| investigating | assigned, rejected              |
| assigned      | resolved, investigating         |

---

# IMPORTANT SYSTEM RULE

Severity and state are separate concepts.

## Severity

Represents environmental danger.

Example:

* Low
* Medium
* High
* Critical

## State

Represents operational progress.

Example:

* reported
* verified
* resolved

The backend must never confuse these concepts.

# REPORT SUBMISSION API

## Endpoint

```text
POST /report
```

## Purpose

Receive new waste reports from frontend.

## Expected Input

*Note: This endpoint accepts `multipart/form-data`. Fields are passed as form fields, and the image as a file.*

```text
Form Data:
  latitude: 9.0579
  longitude: 7.4951
  location_name: "Wuse Market"
  description: "Blocked drainage with waste"
  severity: "High"

File:
  image: [binary_data]
```

## Backend Responsibilities

When a report is submitted:

1. Validate required fields
2. Validate coordinate ranges
3. Store uploaded image
4. Detect possible duplicates
5. Calculate severity score
6. Assign default state = "reported"
7. Generate confidence score
8. Save to database
9. Return structured response

## Expected Response

```json
{
  "success": true,
  "report_id": 12,
  "severity": "High",
  "state": "reported",
  "confidence_score": 85
}
```


## Purpose

Return structured heatmap data for frontend visualization.

## Backend Responsibilities

The backend must:

* aggregate nearby reports
* calculate hotspot intensity

* calculate weights based on severity and density

# HEATMAP ENGINE

The backend heatmap engine is responsible for:

* clustering nearby reports
* calculating intensity values
* preparing frontend-compatible heatmap data
* reducing intensity for resolved incidents
* maintaining dynamic environmental visualization

---

# IMPORTANT HEATMAP RULE

Heatmaps should not permanently count all historical reports.

The backend should prioritize:

* recent reports
* unresolved reports
* high-confidence reports

Old or resolved incidents should gradually lose influence.

---

# REPORT DECAY LOGIC

Older reports should become less influential.

## Example

| Report Age | Influence |
| ---------- | --------- |
| 1 hour     | 100%      |
| 3 days     | 60%       |
| 2 weeks    | 10%       |

This prevents outdated hotspots.

---
